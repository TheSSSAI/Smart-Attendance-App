Description

A modal dialog used to inform the user that a required permission (e.g., location) has been denied and provides a direct path to the device's settings to resolve the issue. This is a critical component for handling permission-related user flows gracefully.

Design System Alignment

This component utilizes the design system's tokens for colors (primary, neutrals, semantic), typography (headings, body text), spacing, and border-radius. It is composed of atomic components like Primary Button and Text.

Responsive Specifications

Mobile Adaptations

  • Dialog takes up most of the screen width with a small margin.
  • Action buttons stack vertically to ensure they are easily tappable.

Tablet Adaptations

  • Dialog maintains its max-width and is centered on the screen.
  • Action buttons remain in a horizontal row.

Desktop Enhancements

  • The overlay provides a clear context shift.
  • The dialog has a subtle entrance animation (scale and fade).

Accessibility Features

Keyboard Support

The dialog, once opened, should trap focus. Users can navigate between the description, 'Cancel' button, and 'Open Settings' button using the Tab key. The Escape key should close the dialog.

Screen Reader Support

The dialog container has `role="dialog"`, `aria-modal="true"`, `aria-labelledby="dialog-title"`, and `aria-describedby="dialog-desc"` to provide context to screen reader users.

Focus Management

When the dialog opens, focus should be programmatically moved to the first focusable element, typically the 'Cancel' button or the dialog container itself.

Aria Attributes

  • role="dialog"
  • aria-modal="true"
  • aria-labelledby
  • aria-describedby

Color Contrast

All text and button colors meet WCAG 2.1 AA contrast requirements.

Implementation Notes

Embedded Css Architecture

The component's CSS is designed to be embedded within a larger stylesheet, using BEM-style naming (`.dialog`, `.dialog__title`) to avoid conflicts. It relies on globally defined CSS custom properties for design tokens.

Component Organization

The HTML structure uses semantic elements where possible. The interactive demo uses the `:target` pseudo-class to toggle visibility without JavaScript, useful for a static showcase.

Responsive Design

A mobile-first approach is used. The base styles are for mobile, and a media query adjusts the layout for larger screens (e.g., button orientation).

Accessibility Compliance

The provided HTML includes necessary ARIA attributes for a modal dialog. A JavaScript implementation would be required to manage focus trapping and the Escape key functionality.

Browser Support

Compatible with modern evergreen browsers. The use of `color-mix()` in the base styles is for demonstrating advanced features and might need fallbacks for older browsers.

Usage Guidelines

When To Use

Use this dialog immediately after a user attempts an action that fails due to a denied permission that they can change in the device settings (e.g., Location, Camera, Notifications).

Content Guidelines

The title should clearly state the required permission (e.g., 'Location Access Required'). The description should concisely explain *why* the permission is necessary for the feature to work. Button text should be clear and action-oriented ('Open Settings', 'Cancel').

Placement Rules

This component is a modal overlay and should appear centered on top of the current screen content, dimming the background to draw focus.

Customization Options

  • The icon, title, and description text are customizable to suit the specific permission being requested (e.g., Camera, Contacts).

Testing Recommendations

Browser Testing

Test on latest versions of Chrome, Firefox, Safari, and Edge.

Device Testing

Verify layout and interaction on both small and large mobile screen sizes to test responsive adaptations.

Accessibility Testing

  • Verify focus is correctly trapped and managed when the dialog is open (requires JS).
  • Test with screen readers (VoiceOver, TalkBack) to ensure the dialog's role, title, and description are announced correctly.
  • Check keyboard navigation with the Tab key.

Performance Testing

Animations should be smooth (60fps) and not cause layout shifts.
