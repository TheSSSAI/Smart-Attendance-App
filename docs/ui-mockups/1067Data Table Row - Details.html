Description

Represents a single row within a data table, containing cells for data display and actions. It is designed to be responsive, transforming into a card-like layout on smaller viewports. It incorporates atomic components like Checkboxes, Badges, and Icon Buttons.

Design System Alignment

This component utilizes the design system's color palette for states (hover, selected), typography for cell content, and spacing tokens for padding and layout. It serves as a container for various atomic components, ensuring a consistent look and feel within larger data structures.

Responsive Specifications

Mobile Adaptations

  • Row transforms into a card with a border and shadow on hover.
  • Each data cell becomes a key-value pair, with the column header shown as a label using the `data-label` attribute.
  • Cells stack vertically within the card.
  • Content within cells is typically right-aligned to align with the labels.

Tablet Adaptations

  • At 768px and above, the card layout transforms into a traditional table row.
  • A CSS Grid is used to align columns, providing more control than a standard HTML table.
  • Cell labels (`data-label`) are hidden.

Desktop Enhancements

  • The grid layout provides ample whitespace and clear column separation.
  • Hover states provide clear feedback.
  • Action buttons are right-aligned for consistent placement.

Accessibility Features

Keyboard Support

The entire row can be made focusable, and interactive elements within (checkbox, buttons) are individually focusable.

Screen Reader Support

The mobile layout's `data-label` provides context for each cell's value, e.g., 'Status: Active'. For desktop, standard table navigation (if using `role` attributes) is supported. Interactive elements have ARIA labels.

Focus Management

Focusable elements like icon buttons and checkboxes have a clear, high-contrast focus ring.

Aria Attributes

  • Using `role='row'` and `role='cell'` on the divs can improve semantics for assistive tech.
  • `aria-label` is used on icon-only buttons for context.

Color Contrast

All text and badge colors are chosen from the design system to meet WCAG 2.1 AA contrast requirements.

Implementation Notes

Embedded Css Architecture

All CSS is self-contained. A mobile-first approach is used, with a media query at 768px to switch to the desktop/table layout. CSS custom properties (`--grid-columns`) are used to define the column layout for each row variant, making it highly maintainable.

Component Organization

The HTML is structured with a parent `data-row` container and child `data-row__cell` elements. Each cell contains a `data-row__cell-content` wrapper to separate the data from the cell's layout properties. This structure supports the responsive transformation.

Responsive Design

The core of the responsive strategy is switching `display` from `block` (mobile card) to `grid` (desktop row) and using the `::before` pseudo-element with `content: attr(data-label)` to display headers on mobile.

Accessibility Compliance

The use of `data-label` attributes is a key accessibility feature for the mobile view, providing context that would otherwise be lost. Standard practices like `focus-visible` and accessible atomic components are used.

Browser Support

The use of modern CSS (Grid, Custom Properties, Flexbox) targets modern evergreen browsers.

Usage Guidelines

When To Use

Use as the primary component for displaying a single record within any `Data Table` or `Approval Queue List` organism.

Content Guidelines

Cell content should be concise. For longer content, consider a detail view. User names and emails should be consistently formatted.

Placement Rules

Should be placed within a container that provides a header row for the desktop view. The header is typically a separate component that is hidden on mobile.

Customization Options

  • Apply a variant class like `.data-row--user` or `.data-row--report` to the main `.data-row` element. This sets the correct CSS grid layout on desktop.
  • Add the `.data-row--selected` class to indicate the row has been selected.

Testing Recommendations

Browser Testing

Test in Chrome, Firefox, Safari, and Edge. Pay close attention to the grid layout rendering.

Device Testing

Test on both mobile and desktop viewports to ensure the responsive transformation from card-to-row is seamless.

Accessibility Testing

  • Verify screen reader output on mobile to ensure `data-label`s are read correctly.
  • Test keyboard navigation for all interactive elements within the row on desktop.
  • Check color contrast of status badges.

Performance Testing

When used in a long virtualized list, verify that rendering a single row is fast and does not cause jank.
